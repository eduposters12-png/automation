SECTION 1 - PROJECT OVERVIEW

What the app does in simple terms:
ListifyAI is a SaaS application for Etsy sellers. It lets a seller register, connect an Etsy shop through OAuth, store a Claude API key, analyze the shop with Claude plus trend search, generate Etsy product listing images with OpenAI image generation, review images with Claude, generate listing copy with Claude, optionally generate a simple listing video from images, and queue completed listings for manual or automatic Etsy upload.

The product workflow is:
1. User creates an account or logs in.
2. User connects Etsy through OAuth.
3. User adds a Claude API key.
4. Backend fetches Etsy shop/listing/sales data and Serper trend data.
5. Claude returns niche, style, strengths, opportunities, and product ideas.
6. User picks or enters a product idea.
7. Backend builds/refines an image prompt, calls OpenAI image generation, uploads the image to Cloudinary, and asks Claude to review the prompt/result concept.
8. User approves images, generates copy, optionally generates a video, and uploads to Etsy or saves the package for manual review.

How frontend and backend communicate:
The frontend communicates with the backend over REST HTTP using fetch. Browser-side calls go through lib/api.ts:18-44. The API base URL is process.env.NEXT_PUBLIC_API_URL with fallback http://localhost:8000 at lib/api.ts:1. Browser requests include credentials with credentials: "include" at lib/api.ts:20-23 so the HTTP-only auth cookie is sent.

Next.js server components call backend APIs through lib/server-api.ts:13-24. That helper reads cookies from next/headers and forwards them as a Cookie header to FastAPI. The server helpers expose getCurrentUser, getDashboardStats, getOnboardingStatus, getSettings, getListings, and getJobs at lib/server-api.ts:26-47.

Authentication method:
Authentication is email/password with bcrypt password hashes and JWT access tokens. Password hashing and verification are in backend/app/core/security.py:17-23. JWT creation and decoding use HS256 in backend/app/core/security.py:25-38. The auth cookie is set in backend/app/api/auth.py:_set_auth_cookie at lines 17-32. The cookie name defaults to listify_session in backend/app/core/config.py:37. Cookie flags are httponly=True, samesite="lax", secure=settings.cookie_secure, and optional domain=settings.cookie_domain.

Protected backend endpoints use get_current_user in backend/app/core/deps.py:25-51. That function accepts either the HTTP-only cookie or an Authorization: Bearer token. Cookie extraction and bearer fallback are in backend/app/core/deps.py:13-22.

Payment method:
Payments use Stripe Checkout subscriptions. The checkout endpoint is POST /stripe/checkout in backend/app/api/billing.py:28-72. It creates a Stripe customer if needed and creates a subscription checkout session with the selected plan price. Stripe webhooks are handled by POST /stripe/webhook in backend/app/api/billing.py:102-154. Stripe price IDs are mapped to plans in backend/app/services/stripe.py:5-36. Subscription cancellation is POST /stripe/cancel in backend/app/api/billing.py:75-91.

SECTION 9 - SECURITY REVIEW

How API keys are stored and encrypted:
Platform keys such as OPENAI_API_KEY, GEMINI_API_KEY, SERPER_API_KEY, Cloudinary keys, Stripe secret/webhook secret, Etsy client ID, JWT secret, and ENCRYPTION_KEY are stored in backend environment variables defined by backend/.env.example.

User Etsy tokens and Claude keys are stored encrypted in the shops table:
shops.etsy_access_token_encrypted.
shops.etsy_refresh_token_encrypted.
shops.claude_api_key_encrypted.

Encryption and decryption are in backend/app/core/security.py:41-87.
The code accepts a 64-character hex ENCRYPTION_KEY or 32 raw bytes.
encrypt_secret uses AES-CBC with a random 16-byte IV and PKCS7 padding. It stores base64url(iv):base64url(ciphertext).
decrypt_secret reverses that process.

Important encryption gap:
AES-CBC encryption is not authenticated. There is no HMAC or AEAD mode such as AES-GCM. This means encrypted values are confidential but not tamper-evident. A stronger implementation should use Fernet, AES-GCM, or AES-CBC plus HMAC.

How auth tokens are managed:
JWTs are created with HS256 in backend/app/core/security.py:25-31.
Tokens contain sub and exp only.
Tokens are stored in an HTTP-only cookie by backend/app/api/auth.py:24-32.
Default cookie name is listify_session.
Cookie expiration defaults to 7 days through access_token_expire_minutes at backend/app/core/config.py:40.
get_current_user reads the cookie or bearer token, decodes it, validates UUID, loads the user, and rejects soft-deleted users at backend/app/core/deps.py:25-51.

Sensitive endpoint protection:
Most user data and state-changing endpoints require get_current_user.
/stripe/webhook has no user auth but verifies Stripe signature, which is correct for webhooks.
/health is public, which is normal.
/auth/logout is public in code, but it only deletes the caller's cookie.
/docs and /redoc are public because FastAPI docs are enabled.

Security strengths:
Passwords are hashed with bcrypt through passlib.
JWTs are in HTTP-only cookies, reducing exposure to JavaScript.
Bearer token fallback exists for API clients.
Etsy OAuth uses PKCE and state cookies.
Etsy and Claude secrets are not returned raw to the frontend.
CORS allows only the configured frontend origin and credentials in backend/app/main.py:19-25.
TrustedHostMiddleware is configured from frontend/backend hostnames in backend/app/middleware/security.py:13-25.
Stripe webhooks verify stripe-signature.
Rate limiting is globally configured with SlowAPI at 60/minute.

Security gaps and risks:
No CSRF token protection exists for cookie-authenticated state-changing requests. SameSite=Lax helps, but a production SaaS should add CSRF protection for POST/PATCH/DELETE endpoints.
cookie_secure defaults to False in backend/app/core/config.py:39. Production must set COOKIE_SECURE=true.
AES-CBC encryption lacks authentication.
JWT tokens have no jti, rotation, server-side invalidation, or logout revocation list. Password changes do not invalidate existing sessions.
Account deletion does not cancel the Stripe subscription or revoke Etsy tokens; it only soft-deletes locally and deletes the cookie.
Stripe checkout errors return str(exc) to clients at backend/app/api/billing.py:69-70 and cancel errors at lines 85-86. This can expose provider details.
The in-memory upload queue is not durable and runs in-process. A server restart loses queued jobs.
Rate limiting is IP-based only, not user-based or endpoint-cost-based. Expensive AI endpoints should have stricter per-user rate limits.
No explicit file size/content-type checks exist when downloading images/videos from URLs for ZIP or Etsy upload. The URLs are generated by the app, but defense-in-depth would still help.
No backend Sentry/error tracking exists, despite many third-party integration points.
Frontend error parsing in lib/api.ts only handles string detail. Structured security/credit errors would currently become "Something went wrong".
The app exposes public /docs and /redoc in backend/app/main.py. That may be acceptable in staging but should be evaluated for production.
Etsy callback depends on the user auth cookie still being present on callback. Cross-domain cookie settings must be correct or OAuth can fail.

Where credit balance should be checked:
Check credit balance before the AI call or before queueing external work. Do not call Claude/OpenAI or start Etsy upload unless the user has enough credits.

Recommended flow:
1. Validate auth, ownership, plan restrictions, and input.
2. Determine action cost.
3. Atomically check and deduct or reserve credits before the provider call.
4. Call the provider.
5. If the provider call fails because of platform/provider error and the user received no value, refund credits in the same service layer with a REFUND ledger entry.
6. If the user caused the failure with invalid input or missing credentials, decide product policy. I recommend no deduction when validation fails before the external call.

How insufficient credits should be returned:
Use HTTP 402 Payment Required. The response detail should be structured, not only a string.
Recommended FastAPI detail:
{
  "code": "INSUFFICIENT_CREDITS",
  "message": "Insufficient credits for image generation.",
  "required": 5,
  "balance": 2,
  "action": "IMAGE_GENERATION"
}

Frontend handling:
lib/api.ts should preserve this object on ApiError, for example by adding detail and code fields. UI components should detect code INSUFFICIENT_CREDITS and show a modal with balance, required credits, and an upgrade/top-up CTA.

Important implementation detail:
Credit deduction must be atomic. Do not do "read balance, then write balance" without locking. Use a transaction with SELECT ... FOR UPDATE on the user row or an atomic UPDATE users SET credit_balance = credit_balance - cost WHERE id = :id AND credit_balance >= :cost RETURNING credit_balance.
